# Architecture

This document describes the design commitments that should guide future
versions of `DiversityAndDissimilarity.jl`. It is written for maintainers and
contributors, not as user-facing tutorial material.

## Purpose

`DiversityAndDissimilarity.jl` provides a compact Julia implementation of
diversity, similarity, dissimilarity, distance, and divergence measures for
ecological and related categorical abundance data.

The package is intentionally built around small dispatchable index objects
rather than a large set of unrelated functions. A value such as `Shannon()`,
`BrayCurtis()`, or `JensenShannon()` carries the convention being requested;
generic operations such as `entropy`, `diversity`, `similarity`, and
`dissimilarity` perform the calculation.

## Goals

- Provide clear, tested implementations of common diversity and pairwise
  comparison indices.
- Make convention choices explicit, especially where multiple communities use
  the same name differently.
- Support common ecological data shapes: abundance vectors, observation
  vectors, dictionaries, community matrices, and Tables.jl-compatible tables.
- Keep the runtime dependency footprint small.
- Preserve predictable row/column semantics: community matrices are samples in
  rows and taxa/categories in columns.
- Expose machine-readable metadata for generic workflows, reports, audits, and
  documentation.
- Support low-count and finite-sample estimator choices where they materially
  affect interpretation.
- Maintain cross-package validation against hand calculations and external
  tools such as vegan, scikit-bio, SciPy, iNEXT, and published examples.
- Make documentation executable where practical through doctests, generated
  assets, and reproducible notes.

## Non-Goals

- This package is not intended to become a full ecological modelling framework.
  Ordination, regression, null models, constrained analysis, and broad
  ecological workflows should remain outside the core unless they are needed to
  validate or explain an index.
- This package should not absorb every index from every ecosystem package
  without clear conventions, tests, and documentation.
- This package should not hide convention differences behind a single string
  API such as `method="..."`.
- This package should not require heavy runtime dependencies for optional
  workflows. DataFrames, plotting, R, Python, Graphviz, LaTeX, and documentation
  tooling belong in tests, docs, validation, scripts, or optional workflows.
- This package should not silently normalize, reorder, or reinterpret data in
  ways that change the scientific meaning of the result.
- This package should not overstate mathematical properties. If boundedness,
  metric status, symmetry, or triangle-inequality behavior is not encoded with
  confidence, return `:unknown` rather than inventing certainty.

## Framework Structure

The public type hierarchy has two main roots:

- `DiversityIndex`: the root for alpha-diversity and pairwise comparison
  indices.
- `ShannonEstimator`: the root for entropy and divergence estimators.

`DiversityIndex` has two direct subfamilies:

- `AlphaDiversityIndex`: one-assemblage indices such as `Richness`, `Shannon`,
  `Renyi`, `Tsallis`, `Simpson`, `GiniSimpson`, `Hill`, `Chao1`, `ACE`,
  `SampleCoverage`, `PielouEvenness`, and `FisherAlpha`.
- `PairwiseIndex`: two-assemblage indices such as `Jaccard`, `BrayCurtis`,
  `Hellinger`, `KullbackLeibler`, `JensenDifference`, `JensenShannon`, and
  related similarity/dissimilarity measures.

The type tree is intentionally shallow. Most behavior should live in methods
and metadata traits rather than deeper inheritance. The generated type-tree
documentation lives in `docs/src/type-structure.md`, and the diagram is
generated by:

```bash
julia --project=docs docs/make_type_trees.jl
```

## Source Layout

- `src/DiversityAndDissimilarity.jl`: module definition, exports, and includes.
- `src/utilities.jl`: input handling, community matrix conversion, counts, and
  proportions.
- `src/diversity.jl`: alpha-diversity indices, entropy estimators, uncertainty
  helpers, bootstrap/jackknife, and alpha summaries.
- `src/similarity.jl`: pairwise similarity, dissimilarity, distance, and
  divergence indices.
- `src/framework.jl`: metadata traits, reference cases, estimator reports, and
  workflow/audit helpers.
- `test/runtests.jl`: unit, reference, identity, bound, and framework tests.
- `test/quality`: Aqua/JET checks that run separately from the main tests.
- `validation`: external reference datasets, manifests, and candidate fixtures.
- `docs`: Documenter manual and generated documentation assets.
- `notes`: LaTeX notes/manuscript materials and downloaded references.

## Core API Principles

The generic operations are the primary public API:

- `entropy(index, data)`: entropy-family values in entropy units.
- `diversity(index, data)`: alpha-diversity values, including effective
  diversity where that transformation is conventional.
- `effective_diversity(index, data)`: explicit effective-diversity
  transformations.
- `similarity(index, left, right)`: pairwise similarity or coefficient forms.
- `dissimilarity(index, left, right)`: pairwise dissimilarity, distance, or
  divergence forms.
- `distance(index, ...)`: alias for dissimilarity/distance workflows.

Convenience functions such as `shannon_entropy`, `bray_curtis_distance`, and
`jensen_shannon_distance` are useful and should remain, but they should call
the generic API or share its implementation. New work should avoid creating a
parallel implementation path for convenience names.

## Data Semantics

The package supports several input styles, each with explicit meaning:

- Numeric vectors are abundance vectors by default.
- Non-numeric vectors are raw observations.
- Numeric vectors can be treated as observations with `frequencies=false`.
- Dictionaries are category-to-abundance maps.
- Matrices are samples in rows and taxa/categories in columns.
- Tables.jl-compatible inputs are converted through `community_matrix`.

Important preservation points:

- Keep row-wise sample semantics for matrices.
- Keep the `frequencies` keyword behavior consistent across alpha and pairwise
  functions.
- Keep `species` explicit for tables with metadata columns.
- Reject invalid abundance data early: negative, non-finite, or all-zero
  abundance vectors should fail clearly.
- Do not use unordered dictionary iteration when alignment matters for pairwise
  comparisons; align categories explicitly.

## Metadata And Traits

Every public index should participate in the metadata system in
`src/framework.jl` where appropriate:

- `index_family`
- `input_mode`
- `output_mode`
- `index_range`
- `index_bounds`
- `is_finite`
- `is_symmetric`
- `is_nonnegative`
- `is_bounded`
- `is_metric`
- `is_triangular`
- `is_pseudometric`
- `is_quasimetric`
- `is_metametric`
- `is_semimetric`
- `is_premetric`
- `is_supermetric`
- `is_similarity`
- `is_dissimilarity`
- `requires_probabilities`
- `supports_matrix_kernel`
- `_index_formula`
- `_index_aliases`
- `_index_notes`

Metadata is part of the package's public promise. When adding or changing an
index, update the metadata, documentation, and tests together. Unknown
properties should return `:unknown`; do not overclaim.

`index_bounds` should describe both numeric bounds and their interpretation.
For similarities, a lower bound usually means no overlap or complete
dissimilarity. For dissimilarities and distances, a lower bound usually means
identical or indistinguishable inputs.

## Estimator Design

Entropy and information-divergence estimators are represented as
`ShannonEstimator` subtypes:

- `Plugin`
- `MillerMadow`
- `HausserStrimmer`
- `Basharin`
- `AddGamma`
- `ChaoShen`

Estimator choices should be explicit objects, not strings. This keeps methods
dispatchable and allows the same estimator vocabulary to be reused across
Shannon entropy, KL divergence, Jensen/Shannon differences, and
Jensen-Shannon divergence or distance.

Low-count corrections should preserve their scientific assumptions in docs and
notes. For example, pseudocount methods require support choices, shrinkage
methods imply a target, and Good-Turing/coverage methods rely on singleton and
unseen-mass logic.

## Similarity, Dissimilarity, And Divergence Conventions

Pairwise indices should clearly state whether their primary form is a
similarity, dissimilarity, distance, coefficient, or divergence.

Important conventions:

- `KullbackLeibler` is asymmetric and can be infinite.
- `JensenDifference` is the Jensen difference; for Shannon entropy it equals
  Jensen-Shannon divergence.
- `JensenShannon(; distance=true)` returns the square-root distance form by
  default.
- Similarity/dissimilarity complements should be documented and tested, not
  assumed.
- Matrix methods for asymmetric indices must not force symmetry.

## Validation Strategy

Validation should combine several layers:

- Small hand-checkable examples in tests.
- Published examples and textbook values where available.
- Cross-package reference cases against vegan, scikit-bio, SciPy, iNEXT, and
  related packages.
- Mathematical identity tests, such as complements, bounds, symmetry, and
  known transformations.
- Generated expected-value fixtures for larger external datasets.

The `validation` directory should hold external dataset candidates and
manifests. CI tests should not depend on R or Python being available unless a
separate optional validation workflow is deliberately added.

## Documentation Strategy

Documentation should be kept in sync with implementation:

- README for quick public orientation.
- `docs/src` for the Documenter manual.
- `docs/src/api.md` for public API links and docstring surfacing.
- `docs/src/type-structure.md` for generated type hierarchy diagrams.
- `docs/src/framework.md` for metadata and trait conventions.
- `notes` for LaTeX notes, references, and manuscript-style explanation.

When a new index, estimator, trait, or convention is added, update all usual
places in the same change: source docstrings, manual pages, README where
appropriate, API reference, tests, validation notes, and notes/manuscript
material if the change affects methodology.

## Performance Principles

- Prefer simple, type-stable loops and small helper functions.
- Avoid unnecessary allocations in matrix kernels and pairwise loops.
- Keep specialized matrix kernels for common high-use indices when they provide
  clear value.
- Do not trade clarity for micro-optimizations in scalar methods unless
  profiling shows a real bottleneck.
- Document scaling behavior for operations that produce pairwise matrices.
- Preserve the ability to compute pairwise results without requiring a full
  downstream modelling stack.

## Error Handling And Numerical Behavior

- Invalid inputs should produce clear `ArgumentError`s.
- Non-finite and negative abundance inputs should be rejected.
- All-zero abundance inputs should be rejected for probability-based
  calculations.
- Logarithm bases should be explicit and tested.
- Infinite values are acceptable when mathematically correct, such as
  unsmoothed KL divergence with zero support in the right-hand distribution.
- Corrected estimators should avoid returning negative divergence values caused
  only by finite-sample correction noise when a nonnegative mathematical
  quantity is intended.

## Compatibility And Dependency Policy

The package should remain small and easy to install. Runtime dependencies
should be added only when they serve the core package, not documentation,
plotting, benchmarking, or optional validation.

Current optional or non-runtime tooling belongs in scoped environments:

- Documentation dependencies in `docs`.
- Aqua/JET dependencies in `test/quality`.
- R/Python reference tooling in validation or benchmark scripts.
- Graphviz for generated documentation assets, invoked by scripts rather than
  required by the package.

## What To Preserve In Future Versions

- Dispatchable index objects as the center of the API.
- Generic operations as the primary calculation surface.
- Explicit conventions over stringly typed method names.
- Shallow type hierarchy with behavior in methods and metadata traits.
- Conservative metadata claims with `:unknown` for unencoded properties.
- Row-wise community matrix semantics.
- Clear distinction between similarity, dissimilarity, distance, coefficient,
  and divergence.
- Separate main tests, quality checks, documentation builds, validation assets,
  and notes.
- Cross-package validation as an ongoing design constraint.
- Documentation that explains scientific interpretation, not only syntax.

## Adding A New Index

When adding a new index, complete this checklist:

1. Add the index type to the appropriate family in `diversity.jl` or
   `similarity.jl`.
2. Implement the generic operation methods first.
3. Add convenience functions only after the generic method is working.
4. Add metadata in `framework.jl`, including family, mode, range, bounds,
   metric-like traits, formula, aliases, and notes.
5. Add tests for formula values, edge cases, metadata, and any matrix method.
6. Add documentation in the relevant manual page and API reference.
7. Add or update validation references when an external package or paper gives
   comparable values.
8. Regenerate generated docs assets if the type tree changed.

This checklist is intentionally repetitive. The package's value comes from
having calculations, conventions, validation, and documentation move together.
